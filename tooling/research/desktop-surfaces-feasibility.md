# Claude Desktop surfaces — monitoring feasibility (2026-06-25)

Generated by the `claude-desktop-monitor-feasibility` workflow (4 research agents + judge), grounded in inspection of this machine.

## Judge — overall
Extend selectively, and only after the flagship Claude Code work ships. Three of the four surfaces collapse to clear verdicts: (a) the Desktop CHAT app is cloud-backed with no local event feed or permission contract - skip it beyond a free 'activate window' button; (b) MCP-as-monitor is architecturally incapable of status/approve-deny/jump - drop it entirely; (c) 'Claude Code via desktop' is mostly a non-event - the native installer and IDE-launched CLI are the SAME ~/.claude hooks + projects surface NotchNerd already owns, so they are already covered (verify, don't build). The ONE genuinely new, worthwhile surface is Claude Cowork / 'local agent mode': because its agent runs on-device in an Apple-VZ + gVisor Linux VM, it leaves a live, readable footprint (local-agent-mode-sessions/local_<uuid>.json + audit.jsonl) that yields real live status via the exact same read-only file-watching pattern NotchNerd already uses for ~/.claude/projects. So: ship a read-only Cowork status watcher (plus activate-the-app jump) as a clearly-labelled beta extension; explicitly do NOT promise approve/deny or per-session jump on any desktop surface. The flagship hook-driven control (live status + approve/deny + terminal jump) remains exclusive to Claude Code the CLI - that asymmetry is real and should be communicated, not papered over.

## What is NOT feasible
Blunt list of what cannot be done locally: (1) APPROVE/DENY on any desktop surface (chat OR Cowork). The decision is owned by cloud (chat) or the desktop app's host loop + React UI (Cowork); there is no host-side hook subprocess to block and write a directive to, so NotchNerd's signature 'hold the hook, write stdout over the socket' mechanism simply does not exist off the CLI. The only way to resolve a desktop prompt is a synthetic Accessibility click against an undocumented Electron/web modal - fragile, TCC-gated, breaks every UI revision; not a real surface. (2) Reliable turn-accurate LIVE STATUS for the chat app - it is cloud-backed; there is no hook/event feed, only heuristic mcp.log (post-approval, MCP-only) + locked leveldb telemetry + AX scraping. (3) Reading chat conversation content locally - IndexedDB has no plaintext role markers; bodies are cloud-synced/encrypted. (4) Per-session / per-conversation JUMP - desktop sessions share one Electron window; claude:// deep-linking to a specific conversation or Cowork session id is undocumented (needs RE) and there is no per-session window to raise. (5) MCP as a monitor - it can never see the session, built-in tools, the Cowork VM's own tools, or the permission prompt; the limit is the protocol, not effort. (6) Hooking the embedded Cowork/CCD Claude Code - CLAUDE_CONFIG_DIR is redirected to a per-session isolated .claude (hooks:null) and tools run inside the VM, so managed hooks never fire and a macOS hook-CLI path cannot resolve in-guest.

## Sequencing
Do not insert any of this into Phases 0-6 except two free verifications. (A) Phase 0 (De-risk preservation): add a one-line regression assertion that managed hooks fire for a native-installer `claude` (~/.local/bin/claude) and an IDE-launched `claude` - this proves 'Claude Code via desktop app' is already covered and costs ~0. (B) Phase 4 (Ghostty terminal jump): optionally read ~/.claude/ide/*.lock workspaceFolders/ideName so an IDE-launched session jumps to the IDE's embedded terminal rather than a standalone window - small, in-scope. (C) The genuinely new Cowork work becomes a NEW Phase 7 (post-Phase-6 'Modernize & harden'), because it rides on a private/undocumented format that Anthropic can break any release and must not be allowed to destabilise the flagship CLI integration. Phase 7 reuses the Phase-2 ClaudeTranscriptDiscovery design as a second watcher root (~/Library/Application Support/Claude/local-agent-mode-sessions/), surfaces a read-only Cowork session chip + activate-app jump, and ships behind a 'Cowork (beta)' Defaults flag with a HookHealthCheck-style format-drift guard. The chat-app and MCP-monitor surfaces are dropped from the roadmap entirely.

## Notes for the plan
Concrete notes for the plan author: 1) 'Claude Code via the desktop app' is NOT new scope - the native installer is just a binary-delivery channel (path:native; same `CLAUDE_CONFIG_DIR ?? ~/.claude` resolver), and IDE-launched is the same HOME=~ CLI; both are already covered by the existing hooks + projects pipeline. Add a verification, not a feature. 2) New Phase 7 = read-only Cowork live-status watcher. Reuse the Phase-2 ClaudeTranscriptDiscovery code as-is with a new root: ~/Library/Application Support/Claude/local-agent-mode-sessions/<accountId>/<orgId>/. Parse local_<uuid>.json (title, model, permissionMode, cwd, lastActivityAt, isArchived, spaceId) for the session chip; tail audit.jsonl (stream-json; HMAC-signed so READ ONLY, never write/tamper) for turn activity + result completion. 3) Hard scope guard: NO approve/deny and NO per-session jump on any desktop surface - observable-not-resolvable; only activate com.anthropic.claudefordesktop. 4) Do NOT build an MCP monitor or chat-app AX automation - both fail the three goals on architecture, not effort. 5) Format-drift is a standing tax (already true for CLI hook schema 9 to ~30 events); gate Phase 7 behind a 'Cowork beta' Defaults flag + schema-version guard so a Claude desktop update degrades gracefully instead of crashing the notch. 6) ps was blocked in research sandbox - prefer file-watching over process scanning for VM liveness (vm_bundles/claudevm.bundle/vmIP mtime, rootfs.img/sessiondata.img). 7) Secrets live in these dirs (MCP X-API-KEY, account tokens, .audit-key) - the watcher must read only session metadata/transcript files, never the key material.

## Per-surface verdicts
### Claude Desktop chat app (macOS)
- monitorable: **partial**  | effort: 0.5-8 wk
- features: jump-to-window
- mechanism: No hook/IPC equivalent exists; the chat agent loop and permission decisions live in the cloud-backed claude.ai web UI rendered inside an Electron shell. Only three local levers exist, in descending robustness: (1) jump-to-window via `open -a Claude` / NSWorkspace activate / the registered `claude://` scheme (stable; per-conversation deep-link undocumented); (2) a HEURISTIC activity feed by tailing ~/Library/Logs/Claude/mcp.log (post-approval MCP tool calls only) fused with Local Storage leveldb telemetry (claudeai.session.response_delayed etc.) - undocumented, locked-while-running, MCP-only; (3) Accessibility scraping of the renderer to read the streaming/thinking indicator and the Allow/Decline modal, plus synthetic AX/CGEvent clicks - the ONLY path to approve/deny and pure-fragile. The official extension surface (MCP/Desktop Extensions) extends Claude's tools, it does not observe the session or intercept the app's own permission prompt.
- fragility: Jump: low-medium (URL-scheme/activation stable; per-conversation deep-link undocumented). Live status: high (mcp.log/leveldb are private, version-pinned formats; leveldb locked while app runs; AX trees shift every release; app auto-updates, already v1.15962.0). Approve/deny: very high (AX synthetic-click against a cloud-driven web modal, zero contract, silently breaks). No first-class local automation contract of any kind.
- **recommendation:** SKIP as a real integration. Conversation content, turn state, and the permission prompt are cloud-backed with no local event feed, and approve/deny is reachable only through brittle AX hacks. The only piece worth taking is the near-free jump-to-window (activate Claude.app), and only if desktop awareness already exists from Cowork support. Do not build heuristic status or AX approve/deny for the chat app - effort buys a strictly worse experience than the CLI hook control NotchNerd already has.

### Claude Cowork (Anthropic desktop local agent mode)
- monitorable: **partial**  | effort: 2-4 wk
- features: live status, jump-to-window
- mechanism: The agent runs ON-DEVICE in an Apple Virtualization.framework + gVisor Linux microVM, leaving a rich, live local footprint. Live status: an FSEvents/DispatchSource watcher rooted at ~/Library/Application Support/Claude/local-agent-mode-sessions/<acct>/<org>/ that parses per-session local_<uuid>.json metadata (title, model, permissionMode, cwd, userSelectedFolders, lastActivityAt, isArchived, spaceId) and tails each session's audit.jsonl (append-only stream-json: user/assistant/system/result/rate_limit_event, HMAC-signed). This is the SAME read-only pattern as NotchNerd's existing ClaudeTranscriptDiscovery (~/.claude/projects/**/*.jsonl), just pointed at a new private directory. Jump: activate com.anthropic.claudefordesktop (all sessions share one window; no per-session window) plus best-effort claude:// deep-link (RE needed). Approve/deny: NO clean mechanism - the gate runs in the desktop app's host loop (hostLoopMode:true) + React UI, governed by bypassPermissionsGateByAccount/orgCliExecPolicies; there is no host-side hook subprocess to block, so the CLI 'hold the hook and write stdout' trick does not translate. You can INFER a pending tool (tool_use with no following result in non-bypass modes) but can only resolve it via in-app/AX click.
- fragility: High. The whole local-agent-mode/cowork/vm_bundles layout is a private, undocumented, versioned internal format (app v1.15962.0, SDK 2.1.187) that Anthropic can rename/restructure any release. audit.jsonl is HMAC-signed so writing/tampering is pointless and detectable (read-only is fine). Live-status watching is moderately robust because the stream-json shape is stable-ish, but it is unsupported. Jump deep-link and approve/deny are very fragile. No sanctioned API.
- **recommendation:** YES, LATER - and ONLY as a read-only live-status watcher (the strong, recommended capability) plus activate-the-app jump. This is the most monitorable of the three desktop targets precisely because the agent executes locally. Explicitly do NOT pursue approve/deny here: it is observable-not-resolvable and would require unsupported in-VM hook injection across the NAT or AX-clicking the app - both dead ends. Treat the watcher as a second root of the existing transcript-discovery code, clearly labelled 'Cowork (beta, may break on Claude updates)'.

### Claude Code via the desktop app (native installer / IDE-launched / embedded CCD)
- monitorable: **full**  | effort: 0-1 wk
- features: live status, approve/deny, jump-to-window
- mechanism: Splits cleanly by CLAUDE_CONFIG_DIR. (1) Native-installer CLI (~/.local/bin/claude -> versions/2.1.193, path:native) and (2) IDE-launched CLI (~/.claude/ide/*.lock, Windsurf/VS Code/JetBrains) both resolve config via `process.env.CLAUDE_CONFIG_DIR ?? join(homedir(),'.claude')` with NO override - they read ~/.claude/settings.json (NotchNerd's managed hooks) and write ~/.claude/projects/**/*.jsonl. The native installer is only a delivery channel, not a different runtime, so NotchNerd's EXISTING hook + transcript pipeline already covers them: live status, approve/deny (blocked-hook-stdout directive), and jump all work with zero/near-zero code. Only refinement: for IDE sessions, read the *.lock workspaceFolders/ideName to focus the IDE's embedded terminal instead of a standalone window. (3) EMBEDDED Cowork/CCD is the SAME engine but spawned with CLAUDE_CONFIG_DIR redirected to a per-session isolated .claude (hooks:null) and tools executing inside the Linux VM, so managed hooks NEVER fire - that path == the Cowork surface (read-only status only).
- fragility: Low for native + IDE CLI - it is the stable, documented hooks + projects surface NotchNerd already depends on and survives version bumps. Medium-high only for the embedded Cowork/CCD slice, which is the private local-agent-mode format covered under the Cowork surface.
- **recommendation:** YES, NOW - but this is a verification, not a build. The native installer and IDE-launched CLI are ALREADY fully covered by the existing hook/transcript integration with zero extra work; just add a Phase-0 regression assertion that hooks fire for a native-installer `claude` and an optional IDE-lock-aware jump target in Phase 4. Do NOT try to hook the embedded Cowork/CCD path - its redirected config dir + in-VM execution make the hook mechanism inapplicable; route it to the read-only Cowork watcher instead.

### MCP-as-monitor for the desktop app
- monitorable: **no**  | effort: 0.5-1 wk
- features: 
- mechanism: Architecturally incapable of the three goals. A self-authored MCP server sees ONLY calls to its OWN tools (name + args + its own result) and can ping NotchNerd's socket on each - but the model must choose to call it, so it is an intermittent, gameable activity light, not a status feed. It cannot see the conversation, user prompts, model output, built-in tools, the Cowork VM's own bash/file tools, or other servers. The permission/approve dialog fires in the client BEFORE the server is invoked and a denial never reaches the server, so MCP has no message to observe it and no way to answer it (elicitation is the inverse - the server asking the user). The server is headless/window-agnostic, so no jump. Tailing ~/Library/Logs/Claude/mcp.log gives broader (all-server) but still MCP-only, undocumented, post-approval visibility. On this build the client advertised only the MCP-UI capability - no sampling/elicitation/roots flowed.
- fragility: High for the stated goals, and the ceiling is protocol-architectural, not implementation effort - more engineering cannot unlock approve/deny or true status. mcp.log is an undocumented internal format; coverage is fundamentally incomplete; Cowork's in-VM tools never traverse user-side MCP at all.
- **recommendation:** NO. Do not build an MCP server (or .mcpb) as a monitoring channel - it delivers none of live status, approve/deny, or jump. MCP is for ADDING tools, not observing sessions. At most, mcp.log tailing is a weak supplementary tool-activity signal, and only relevant to the chat-app path that is itself recommended-skip. Treat as a non-starter.

### Claude Desktop chat app (macOS)
- runsLocallyOrCloud: Cloud-backed GUI. The agent/chat reasoning runs on Anthropic servers (claude.ai); the desktop app is a local Electron renderer of that web app. The only things that genuinely execute locally are (1) MCP servers / Desktop Extensions the user installs (spawned as local child processes by the app, e.g. npx mcp-remote, the Filesystem extension), and (2) the Electron host process itself. Conversation history, turn/streaming state, and tool-permission decisions are managed in the renderer + server, not in a local agent loop.
- extensionSurface: Official local extension surface = MCP only, in two equivalent forms: (1) classic local MCP servers declared in ~/Library/Application Support/Claude/claude_desktop_config.json (mcpServers{ name: {command,args,env} }), and (2) Desktop Extensions / Connectors — .mcpb (formerly .dxt) one-click-installable bundles that ARE packaged MCP servers (manifest.json + bundled runtime; secrets stored in OS keychain; installed under "Claude Extensions/"). There are NO hooks for the chat app (the hooks/settings.json event system is exclusive to Claude Code the CLI), and there is NO official permission-callback, observer, or IPC API. Critically, MCP/extensions extend Claude's TOOLS (things Claude can call); they are not a passive monitor of the chat and cannot observe arbitrary session activity or intercept the app's tool-approval prompts. The claude:// URL scheme is registered (deep-link/launch) but is not a documented automation API.
- permissionPromptObservable: observable-not-resolvable

WHY MCP DOESN'T SOLVE THE THREE GOALS: an MCP server/extension is a TOOL Claude can call; it sees only its own invocations, cannot watch the broader chat, and cannot intercept the app's own Allow/Decline prompt (that prompt gates whether the MCP tool runs at all). So you can't repurpose MCP as a monitor of session status or as a permission relay.

OBSERVABILITY DETAIL (the load-bearing finding): ~/Library/Logs/Claude/mcp.log logs JSON-RPC both directions, including the full tool call, e.g. `[Filesystem] Message from client: {"method":"tools/call","params":{"name":"edit_file","arguments":{"edits":[...],"path":"/Users/hamza/knowledge-base/hot.md"}},"jsonrpc":"2.0","id":15}`. This is a real, tailable local signal of tool activity WITH arguments. BUT: it only fires for configured MCP servers, and it logs the call only AFTER the user approves it — a Decline yields no MCP traffic at all. So mcp.log reflects resolved approvals, never the pending prompt. The pending tool-permission modal lives purely in the renderer/web UI and is not written to any local log (main.log's 'Blocked permission check {permission: background-sync}' lines are Chromium web-permission noise, unrelated to tool approvals). Local Storage leveldb carries a live-ish telemetry stream (claudeai.session.response_delayed, thinking_cell.clicked, perf.interaction_histogram) with session_id/connection_state/seconds_since_last_activity that could be mined for coarse 'busy/stalled' status, but it's an undocumented schema and the DB is locked while the app runs.

DRIVING THE THREE NOTCHNERD CAPABILITIES:
- Jump-to-window: easy and robust — `open -a Claude` / NSWorkspace activate / claude:// ; deep-link to a specific conversation is not a documented capability.
- Live status: only heuristic — fuse mcp.log tail (tool activity) + leveldb telemetry + AX reading of the thinking/stop indicator. No clean event feed like Claude Code hooks.
- Approve/deny: no supported path. The modal is observable+clickable ONLY via Accessibility (read the Allow once/Allow always/Decline buttons, synthesize a click). That is fragile (AX trust prompt, breaks on every UI revision, races with streaming). I classified permissionPromptObservable as observable-not-resolvable: observable+resolvable in practice only through brittle AX scraping, with zero official/stable contract.

ADJACENT (out of scope here, other sub-questions): this same app directory also hosts Cowork (cowork_*.log, coworkd.log, cowork_host_loop_debug.log, local-agent-mode-sessions/, bridge-state.json with remote/local session IDs, cowork-enabled-cli-ops.json) and a desktop-bundled Claude Code (claude-code/2.1.187/claude.app, claude-code-vm/, claude-code-sessions/, main.log '[HostLoop] cli.js stderr ...'). WHAT A SERVER CAN/CANNOT SEE (official server-concepts + client-concepts):
- Server primitives: Tools (model-controlled, invoked via tools/call), Resources (application-pulled, read-only), Prompts (user-invoked). A server is reactive: it only ever receives requests addressed to it. It has NO view of the conversation, the model's output, the user's text, OR other servers/built-in tools. This is the hard limit: "a server only sees calls to ITS tools."
- Permission model: per spec, "Tools may require user consent prior to execution"; the host shows the approval dialog and the call only reaches the server AFTER approval. Therefore the server never observes a pending or denied prompt and cannot answer it. There is no MCP message that exposes the client's permission UI to a server.

ADVANCED MCP FEATURES vs MONITORING (client-concepts + logging spec):
- Sampling (sampling/createMessage): SERVER->client request for an LLM completion, client-gated/human-in-loop. Lets a server borrow the model; does NOT reveal the session. Not advertised by this build.
- Elicitation (elicitation/create): SERVER->client request for structured user input (a form). Server-initiated, only while one of its tools runs; it is the server asking the user, NOT a way to intercept Claude's own tool-permission prompt. (Anthropic was still rolling elicitation out across clients in 2025; Claude Code feature-requests #2799/#7108.)
- Roots (roots/list, roots/list_changed): client->server advisory directory scope. Irrelevant to monitoring.
- Notifications/logging (notifications/message, notifications/progress, .../list_changed): flow server->CLIENT. A server can EMIT logs to the client; it cannot RECEIVE session events. The client is free to display/persist them but does not relay them to other apps. So notifications are an outbound, not an inbound, monitoring channel.

CAN A SERVER NOTIFY NOTCHNERD / ANSWER A DECISION?
- Notify: yes, trivially — your server is your own process; on a tools/call it can open a Unix socket to NotchNerd. But it only fires when the model calls that specific tool; you cannot poll the session.
- Answer a permission decision: no — see permission model above.
