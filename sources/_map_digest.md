# DUAL-CODEBASE MAP DIGEST (detailedNotes stripped)

## ============ boringNotch ============
### App entry, lifecycle, and the notch overlay window/panel system

**Summary:** boring.notch is an agentless menu-bar macOS app whose entire UI is a borderless floating NSPanel pinned to the top-center of a screen, drawn over (and shaped like) the physical notch. A SwiftUI `@main App` (`DynamicNotchApp`) installs an `AppDelegate` that builds, positions, shows/hides and multiplexes these panels across displays. Runtime open/closed state lives in per-window `BoringViewModel` instances (`notchState`), while a global singleton `BoringViewCoordinator` owns which inner view (`home`/`shelf`) is shown plus transient HUD/sneak-peek state. `ContentView` is the panel's root SwiftUI view and implements hover-to-open, tap/gesture open, and auto-close with spring animations.

**Data flow:** 1) Launch: `DynamicNotchApp` (@main, boringNotchApp.swift:15-50) declares only a `MenuBarExtra` scene (Settings/Updates/Restart/Quit) and bridges to `AppDelegate` via `@NSApplicationDelegateAdaptor`. `init()` creates the Sparkle `SPUStandardUpdaterController`. The app is effectively agent/accessory (no main window scene). 2) `AppDelegate.applicationDidFinishLaunching` (282-439) registers all observers (screen-params change, custom Notification.Names, lock/unlock distributed notifications), wires global `KeyboardShortcuts` (.toggleSneakPeek, .toggleNotchOpen), then if `!Defaults[.showOnAllDisplays]` creates ONE window on `NSScreen.main` via `createBoringNotchWindow` and calls `adjustWindowPosition(changeAlpha:true)`; otherwise `adjustWindowPosition` builds one window per screen. Finally `setupDragDetectors()`, optional onboarding window / welcome sound, and caches `previousScreens`. 3) Window creation (`createBoringNotchWindow`, 234-265): builds a `BoringNotchSkyLightWindow` (NSPanel) sized `windowSize` (640 x 210), styleMask `[.borderless,.nonactivatingPanel,.utilityWindow,.hudWindow]`, sets its `contentView` to an `NSHostingView(rootView: ContentView().environmentObject(viewModel))`, calls `orderFrontRegardless()`, and inserts the window into `NotchSpaceManager.shared.notchSpace.windows` (the private max-level CGS Space) so it floats above fullscreen apps and all spaces. 4) Positioning (`positionWindow`, 267-280): origin = `screen.frame` top-center: `x = origin.x + width/2 - window.width/2`, `y = origin.y + height - window.height` so the panel hugs the top edge centered over the physical notch; `changeAlpha` fades it in. 5) State at runtime: `BoringViewModel.notchState` is `.closed`/`.open`. `ContentView` reads `vm.notchState` to size/animate and to switch between the closed live-activity strip and the open `home`/`shelf` content (driven by `coordinator.currentView`). 6) Open paths: hover (ContentView.handleHover -> after `minimumHoverDuration` -> doOpen), tap (`onTapGesture{doOpen()}`), swipe-down pan gesture, drag-enters-notch (DragDetector -> vm.open + currentView=.shelf), drop targeting (anyDropZoneTargeting), keyboard shortcut `.toggleNotchOpen`. `doOpen` runs `vm.open()` inside an interactive spring. `vm.open()` sets `notchSize = openNotchSize`, `notchState = .open`, and force-updates MusicManager. 7) Close paths: hover-exit (100ms debounce -> vm.close), swipe-up gesture, drop-debounce, sharing-finished, battery-popover-dismiss, keyboard shortcut, or the 3s auto-close Task from `.toggleNotchOpen`. `vm.close()` resets `notchSize` to `getClosedNotchSize`, sets `.closed`, clears sneakPeek/edge flags, and picks `currentView` (shelf if shelf non-empty & openShelfByDefault, else home unless openLastTabByDefault). Close is BLOCKED while `SharingStateManager.shared.preventNotchClose`. 8) Reactivity: `adjustWindowPosition` is re-invoked on Notification.Name.selectedScreenChanged / notchHeightChanged / showOnAllDisplaysChanged / automaticallySwitchDisplayChanged and on `screenConfigurationDidChange` (which diffs `previousScreens` by count/UUID/frame, then cleans up and rebuilds). Fullscreen hiding flows from `FullscreenMediaDetector.fullscreenStatus[screenUUID]` -> `BoringViewModel.setupDetectorObserver` -> `hideOnClosed` -> `effectiveClosedNotchHeight` collapses to 0.

**Integration notes:** To add a NEW inner panel/tab to the host: (1) add a case to `NotchViews` in boringNotch/enums/generic.swift; (2) add a matching `case` in `ContentView.NotchLayout()`'s `switch coordinator.currentView` (ContentView.swift ~347-352) returning your SwiftUI view, mirroring `NotchHomeView`/`ShelfView`; (3) drive selection by setting `BoringViewCoordinator.shared.currentView = .yourCase` (and optionally `vm.open()` to expand). Your view is rendered only when `vm.notchState == .open`. For a transient closed-state notification/HUD instead, reuse `coordinator.toggleSneakPeek(...)` or `toggleExpandingView(...)` and add rendering branches in the closed-state portion of `NotchLayout()`. The view receives `@EnvironmentObject var vm: BoringViewModel` and `@ObservedObject coordinator/musicManager/...` singletons. Header/tab chrome is `BoringHeader` (components/Notch). New global open/close triggers should call `vm.open()` / `vm.close()` and respect `SharingStateManager.shared.preventNotchClose`. New window-affecting settings should post one of the `Notification.Name` toggles and be handled in `applicationDidFinishLaunching`.

To EXTRACT/port the overlay-window capability itself you must lift, in order: (a) boringNotch/private/CGSSpace.swift (private CoreGraphics/SkyLight Space wrapper, the load-bearing trick that keeps the panel above fullscreen and on all spaces) and managers/NotchSpaceManager.swift; (b) components/Notch/BoringNotchSkyLightWindow.swift — depends on the external `SkyLightWindow` SPM package (`SkyLightOperator.delegateWindow`) plus a manual `dlsym` of `SLSRemoveWindowsFromSpaces`; if you don't need lock-screen display you can instead port the simpler BoringNotchWindow.swift which has zero external deps; (c) the geometry layer sizing/matters.swift + extensions/NSScreen+UUID.swift (+auxiliaryTopLeftArea/safeAreaInsets usage) for notch sizing/positioning; (d) the AppDelegate window lifecycle methods createBoringNotchWindow/positionWindow/adjustWindowPosition/cleanupWindows/screenConfigurationDidChange and the Notification.Name extension. The state machine (BoringViewModel + BoringViewCoordinator + NotchState/NotchViews enums + animations/drop.swift) is largely self-contained but BoringViewModel pulls in MusicManager, WebcamManager, SharingStateManager, ShelfStateViewModel, FullscreenMediaDetector (which needs the `MacroVisionKit` package) and Defaults keys — trim these for a minimal port.

### Expanded-notch feature/tab system (panel routing, tab bar, per-tab state, settings, and closed-notch live activities) in boring.notch

**Summary:** The expanded notch renders exactly one "panel" chosen by a tiny enum NotchViews (.home, .shelf). The selected case lives in the singleton BoringViewCoordinator.currentView and is switched on in ContentView.NotchLayout() under `if vm.notchState == .open`. A static `tabs` array drives the tab bar (TabSelectionView) shown in BoringHeader. Each feature is a SwiftUI View backed by a shared ObservableObject view model (e.g. ShelfStateViewModel.shared), gated by a Defaults key, with its own Settings sidebar section. Closed-notch transient indicators ("Live activities") are a separate system driven by coordinator.expandingView / coordinator.sneakPeek (typed by SneakContentType) and rendered in the closed-state branch of ContentView.NotchLayout().

**Data flow:** 1) BoringViewCoordinator.shared is a @MainActor singleton holding `@Published var currentView: NotchViews` (BoringViewCoordinator.swift:53), default .home. 2) The tab bar TabSelectionView (TabSelectionView.swift:22) iterates the static `tabs` array; tapping a TabButton runs `withAnimation(.smooth){ coordinator.currentView = tab.view }` (line 30). 3) The tab bar is mounted only inside BoringHeader (BoringHeader.swift:20), itself rendered only when `vm.notchState == .open` (ContentView.swift:295-298). The whole tab bar is gated by `(!shelf.isEmpty || alwaysShowTabs) && Defaults[.boringShelf]` (BoringHeader.swift:19). 4) The actual panel body is chosen in ContentView.NotchLayout() (ContentView.swift:345-362): inside `if vm.notchState == .open`, a `switch coordinator.currentView` maps .home->NotchHomeView(albumArtNamespace:) and .shelf->ShelfView(); the surrounding VStack applies the scale+opacity transition. 5) Each panel view reads its own shared ObservableObject (e.g. ShelfView -> ShelfStateViewModel.shared, ShelfSelectionModel.shared) plus the @EnvironmentObject BoringViewModel injected at ContentView.swift:216. 6) When the notch closes, BoringViewModel.close() (BoringViewModel.swift:212-218) and coordinator logic may reset currentView to .home/.shelf; if openLastTabByDefault is true the tab is preserved. 7) SEPARATE PATH for closed-notch Live activities: managers call coordinator.toggleExpandingView(status:type:value:) (BoringViewCoordinator.swift:262, e.g. MusicManager.swift:590) or toggleSneakPeek(...) (line 208, e.g. VolumeManager.swift:43). These set coordinator.expandingView / coordinator.sneakPeek (typed by SneakContentType). ### State Management, Settings, Persistence & Notifications

**Summary:** boring.notch persists user preferences almost entirely through sindresorhus/Defaults (typed `Defaults.Keys` declared in boringNotch/models/Constants.swift, read in views via the `@Default` property wrapper and `Defaults.Toggle`). A handful of UI/runtime flags use SwiftUI `@AppStorage` inside the `BoringViewCoordinator` singleton. Global state is held in a few `ObservableObject` singletons (BoringViewCoordinator.shared, MusicManager.shared, BatteryStatusViewModel.shared, etc.) plus per-window `BoringViewModel` instances injected as `@EnvironmentObject`. The settings window is a single `NavigationSplitView` (SettingsView.swift) hosted in an AppKit `SettingsWindowController`; there is no OS notification (UNUserNotification) system — events surface through an in-notch "sneak peek" / "expanding view" HUD driven by the coordinator, plus a Sparkle auto-updater.

**Data flow:** Persistence read/write (the dominant path): A setting is declared once in boringNotch/models/Constants.swift as e.g. `static let showCalendar = Key<Bool>("showCalendar", default: false)` inside `extension Defaults.Keys` (lines 71-202). A view reads/writes it three interchangeable ways: (1) property wrapper `@Default(.showCalendar) var showCalendar` (SettingsView.swift:712) which gives a SwiftUI Binding; (2) the library's `Defaults.Toggle(key: .showCalendar){...}` convenience (SettingsView.swift:719); (3) imperative `Defaults[.showCalendar]` reads/writes anywhere (e.g. BoringViewModel.swift:214). Changes persist to UserDefaults automatically and propagate reactively to every `@Default` view binding.\n\nSettings-change side effects: Some toggles, after writing the Default, post a classic NotificationCenter notification so the AppKit layer can react (the window geometry isn't derived reactively). E.g. ### Build system, packaging/signing, and the Music subsystem boundary of boring.notch (boringNotch / "dynamicNotch")

**Summary:** boring.notch is a sandboxed macOS 14+ SwiftUI menu-bar app (bundle id theboringteam.boringnotch, marketing 2.7.3 / build 271, Swift 5.0) built from a single Xcode 16 project with two native targets: the main app and a non-sandboxed XPC service (BoringNotchXPCHelper) that performs privileged work (Accessibility, keyboard/screen brightness via private frameworks). It pulls 11 SPM packages (Sparkle pinned 2.9.1 for auto-update, plus Defaults, KeyboardShortcuts, Lottie, etc.) and bundles a local MediaRemoteAdapter.framework + perl adapter to read "now playing" from the private MediaRemote framework. CI signs with "Apple Development"/development export (NOT Developer ID, NO notarization), packages a DMG via dmgbuild, and publishes a Sparkle EdDSA-signed appcast + Homebrew cask. The music subsystem lives almost entirely under boringNotch/MediaControllers + MusicManager and must be preserved intact.

**Data flow:** MUSIC NOW-PLAYING FLOW: (1) On startup MusicManager (boringNotch/managers/MusicManager.swift) reads Defaults[.mediaController] (MediaControllerType in Constants.swift:45) and calls createController(for:) which instantiates one of NowPlayingController / AppleMusicController / SpotifyController / YouTubeMusicController, all conforming to MediaControllerProtocol. (2) Before using NowPlaying, MediaChecker.checkDeprecationStatus() spawns /usr/bin/perl mediaremote-adapter.pl <MediaRemoteAdapter.framework path> <TestClient> test; exit code 1 => NowPlaying deprecated on this OS, MusicManager falls back to appleMusic. (3) NowPlayingController.init dlopens /System/Library/PrivateFrameworks/MediaRemote.framework to grab MRMediaRemoteSendCommand/SetElapsedTime/SetShuffleMode/SetRepeatMode function pointers (used for transport control). (4) setupNowPlayingObserver() launches /usr/bin/perl mediaremote-adapter.pl <frameworkPath> stream as a subprocess; its stdout pipe is read by actor JSONLinesPipeHandler, decoding newline-delimited NowPlayingUpdate/NowPlayingPayload JSON. (5) Each update mutates @Published playbackState (PlaybackState); MusicManager subscribes via playbackStatePublisher (Combine) and republishes @Published songTitle/artistName/albumArt/isPlaying for the SwiftUI views (NotchHomeView, MusicControlButton). Apple Music/Spotify controllers instead drive state through AppleScriptHelper; YouTube Music via its networking layer. The MediaRemoteAdapter.framework, mediaremote-adapter.pl and TestClient are copied into the app bundle (PrivateFrameworks / Resources) by the project's Embed Frameworks + Resources build phases.\n\nXPC/PRIVILEGED FLOW: The app is sandboxed, so privileged work is delegated. XPCHelperClient.shared (boringNotch/XPCHelperClient) opens NSXPCConnection(serviceName: theboringteam.boringnotch.BoringNotchXPCHelper) wrapped by AsyncXPCConnection.RemoteXPCService<BoringNotchXPCHelperProtocol>. The embedded, NON-sandboxed BoringNotchXPCHelper.xpc (Contents/XPCServices) answers AX-trust queries (AXIsProcessTrusted), keyboard backlight (private CoreBrightness KeyboardBrightnessClient) and screen brightness (private DisplayServices + IOKit). VolumeManager/BrightnessManager in the app call through this client.\n\nAUTO-UPDATE FLOW: boringNotchApp.init creates SPUStandardUpdaterController(startingUpdater:true). Sparkle reads SUFeedURL (GitHub Pages appcast.xml) and verifies items with SUPublicEDKey. Because the app is sandboxed, Sparkle's Downloader/InstallerLauncher XPC services are enabled (SUEnable* + the -spks/-spki mach-lookup entitlements). ## ============ openIsland ============
### Open Island — Claude Code Session Detection Engine (hooks → Unix socket bridge → SessionState → SwiftUI)

**Summary:** Open Island detects what Claude Code is doing in real time by installing managed hook commands into ~/.claude/settings.json. Every hook event spawns the `OpenIslandHooks` CLI, which decodes the hook JSON from stdin into a `ClaudeHookPayload`, enriches it with terminal/TTY/Warp context, and ships it over a Unix domain socket (NDJSON envelope) to the in-app `BridgeServer`. The server translates the payload into a tool-agnostic `AgentEvent`, applies it to a `SessionState` reducer, and broadcasts it back over the same socket to an in-process `LocalBridgeClient` observer that `AppModel` drains into its `@Observable` state, refreshing the notch UI. Interactive hooks (PermissionRequest) block the CLI until the user resolves them in the UI, at which point a `ClaudeHookDirective` flows back as the hook's stdout. Hooks are the live realtime path; JSONL transcript discovery + a persisted registry + ps/lsof process discovery are the startup-restore and liveness-fallback paths.

**Data flow:** END-TO-END LIVE PATH (Claude Code → UI):
1) Claude Code reaches a hook point (e.g. PreToolUse) and reads ~/.claude/settings.json `hooks.PreToolUse[].hooks[].command`, which is `'/Users/<you>/Library/Application Support/OpenIsland/bin/OpenIslandHooks' --source claude` (ClaudeHookInstaller.hookCommand, ClaudeHookInstaller.swift:66-68). It spawns that command and pipes the hook event JSON to its stdin.
2) OpenIslandHooksCLI.main() (OpenIslandHooksCLI.swift:32) optionally early-returns if HookSkipConfiguration.shouldSkipHooks (line 36), reads stdin (line 40), decodes JSONDecoder→ClaudeHookPayload (line 70), enriches with `.withRuntimeContext(environment:)` (ClaudeHooks.swift:947 — infers terminal app from TERM_PROGRAM/env, resolves TTY via /usr/bin/tty + /bin/ps, resolves Warp pane UUID via SQLite, runs osascript locators for iTerm/Ghostty/Terminal), and sets payload.hookSource = "claude" (line 73).
3) The CLI builds BridgeCommandClient(socketURL: BridgeSocketLocation.currentURL()) (line 49) and calls client.send(.processClaudeHook(payload), timeout:) (line 79). Socket = OPEN_ISLAND_SOCKET_PATH env, else ~/Library/Application Support/OpenIsland/bridge.sock (BridgeTransport.swift:13-32). Timeout is 24h for PermissionRequest, else 45s (lines 75-77). BridgeCommandClient (BridgeCommandClient.swift) opens AF_UNIX/SOCK_STREAM, sets SO_RCVTIMEO/SO_SNDTIMEO, writes one NDJSON line `{"type":"command","command":{"type":"processClaudeHook","claudeHook":{...}}}`, then blocks reading for a `.response`.
4) BridgeServer (started by AppModel.swift:1104) is listening on bridge.sock AND legacy /tmp/open-island-<uid>.sock (BridgeServer.swift:100-111). Its accept DispatchSource creates a ClientConnection, immediately sends `{"type":"hello",...}` (line 254), and reads/decodes NDJSON via BridgeCodec.decodeLines (BridgeTransport.swift:321). The `.command(.processClaudeHook(payload))` routes to handle() → handleClaudeHook (BridgeServer.swift:460,614).
5) handleClaudeHook switches on payload.hookEventName (BridgeServer.swift:629). Non-interactive events call emit(AgentEvent) and immediately reply `.response(.acknowledged)`. emit (line 2564) does localState.apply(event) then broadcast([.event(event)]) to all connected clients. Examples: .sessionStart→sessionStarted(initialPhase:.completed, isHookManaged via origin=.live); .userPromptSubmit/.preToolUse/.postToolUse/.subagentStart→activityUpdated(phase:.running); .stop/.stopFailure/.sessionEnd→sessionCompleted; metadata/jumpTarget kept in sync via synchronizeClaudeMetadata/synchronizeClaudeJumpTarget.
6) INTERACTIVE events do NOT ack immediately. .permissionRequest (line 715): emits permissionRequested (or questionAsked if toolName==AskUserQuestion) and stores PendingClaudeInteraction{clientID, .permission/.question} in pendingClaudeInteractions keyed by sessionID; the hook's socket stays open and blocked (24h timeout).
7) The broadcast `.event` envelope travels back over the SAME socket to the app's in-process observer: AppModel's LocalBridgeClient (connected in AppModel.connectBridgeObserver, AppModel.swift:1118-1133, registered role .observer at line 1147). LocalBridgeClient.readAvailableData yields each `.event` into its AsyncThrowingStream (LocalBridgeClient.swift:129-133).
8) AppModel's bridgeTask loop `for try await event in stream { applyTrackedEvent(event) }` (AppModel.swift:1164-1165). applyTrackedEvent (line 1461) calls state.apply(event) (line 1486). state is @Observable with a didSet that calls bridgeServer.updateStateSnapshot(state) (AppModel.swift:47-53) so the server's localState stays authoritative. The SwiftUI views observing AppModel.state re-render — notch shows running/needs-approval/etc. It also forwards to Watch/iPhone relay and schedules a notification surface.
9) RESOLUTION (user clicks Allow/Deny in the notch): AppModel.send(.resolvePermission(sessionID, resolution) or .answerQuestion) via bridgeClient (AppModel.swift:1437-1446). BridgeServer.handle(.resolvePermission) (line 330) finds the pendingClaudeInteraction and calls resolvePendingClaudeInteraction (line 2402): builds a ClaudeHookDirective (.permissionRequest(.allow(updatedInput,updatedPermissions)) or .deny(message,interrupt)), emits activityUpdated/sessionCompleted, and send(.response(.claudeHookDirective(directive))) to the ORIGINAL hook's clientID (line 2466).
10) The still-blocked OpenIslandHooks CLI receives that `.response`, ClaudeHookOutputEncoder.standardOutput encodes it (ClaudeHooks.swift:636) to a JSON line like {"continue":true,"suppressOutput":true,"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"}}} and writes it to stdout (OpenIslandHooksCLI.swift:84-86). Claude Code reads that stdout and allows/denies the tool. FAIL-OPEN: if the bridge is down, client.send returns nil, the CLI logs to stderr and exits 0 with no stdout, so Claude Code proceeds unchanged (OpenIslandHooksCLI.swift:79-82,111-114).

STARTUP / FALLBACK PATH: On launch SessionDiscoveryCoordinator.loadStartupDiscoveryPayload (SessionDiscoveryCoordinator.swift:83) runs off-main: loads ClaudeSessionRegistry (persisted), runs ClaudeTranscriptDiscovery.discoverRecentSessions (~/.claude/projects/**/*.jsonl), and locates the hooks binary. applyStartupDiscoveryPayload merges these (mergeDiscoveredSessions, dedup by id or transcriptPath) as completed/stale sessions. ActiveAgentProcessDiscovery (ps -Ao + lsof per pid) finds live claude processes, their cwd/TTY/transcript/terminal app, feeding SessionState.markProcessLiveness so non-hook sessions appear/disappear and hook-managed sessions get a liveness backstop (2 missed polls ⇒ ended) when SessionEnd never arrives.

### Open Island — Permission Approvals, Agent Events, and Notifications/Watch Relay

**Summary:** When an agent (Claude Code, Codex, Cursor, OpenCode, Gemini) hits a permission/question gate, its hook config invokes the OpenIslandHooks CLI, which forwards the hook payload over a Unix-domain socket to the in-app BridgeServer and then BLOCKS on the same socket waiting for a reply (up to a 24h timeout). BridgeServer emits a `permissionRequested`/`questionAsked` AgentEvent that flows into AppModel/SessionState and pops a notification card in the notch. When the user clicks Allow/Deny, AppModel sends a `resolvePermission` command back to BridgeServer, which writes a per-agent "directive" response back down the still-open hook socket; the hook process prints that directive as JSON to its stdout, which the agent reads as its permission decision. There is NO keystroke injection and NO response file for approvals — approval is delivered purely via the blocked hook process's stdout. An optional Apple Watch/iPhone relay (WatchNotificationRelay + WatchHTTPEndpoint, an embedded Bonjour HTTP/SSE server) mirrors these actionable events to a paired phone and lets the user approve/deny remotely; it is off by default and gated behind a setting.

**Data flow:** PERMISSION ROUND-TRIP (the core mechanism):

1. Agent gate fires a hook. The agent's config (installed by OpenIsland*HookInstaller) runs the OpenIslandHooks CLI binary with `--source <agent>`, piping the hook event JSON to stdin (Sources/OpenIslandHooks/OpenIslandHooksCLI.swift:40).

2. Hook forwards + blocks. The CLI decodes the payload and calls `BridgeCommandClient.send(.processClaudeHook(payload), timeout:)`. For interactive events (`permissionRequest`) the timeout is 24h (interactiveClaudeHookTimeout, OpenIslandHooksCLI.swift:6,75-79; Codex uses managedInteractiveTimeout). BridgeCommandClient opens a fresh AF_UNIX SOCK_STREAM connection to the bridge socket (~/Library/Application Support/OpenIsland/bridge.sock, BridgeTransport.swift:13-15), writes the command line, then sits in a blocking read() loop until a `.response` envelope arrives or SO_RCVTIMEO trips (BridgeCommandClient.swift:63-91). THIS BLOCKED PROCESS IS THE AGENT'S OWN HOOK SUBPROCESS — the agent is paused inside it.

3. Bridge ingests + emits an event. BridgeServer.readAvailableData → handle(.processClaudeHook). For Claude's `.permissionRequest` case (BridgeServer.swift:715-761) it emits `AgentEvent.permissionRequested(PermissionRequested(... PermissionRequest(title, summary, affectedPath, primaryActionTitle:"Allow Once", secondaryActionTitle:"Deny", suggestedUpdates) ...))` and — crucially — stashes the waiting hook's connection: `pendingClaudeInteractions[sessionID] = PendingClaudeInteraction(clientID: clientID, kind: .permission(payload))`. It does NOT send a response yet, so the hook stays blocked. (Codex uses `pendingApprovals[sessionID] = PendingApproval(clientID,...)` at BridgeServer.swift:542/570; OpenCode `pendingOpenCodeInteractions` L1117; Cursor `pendingCursorInteractions`.)

4. Event → state → notch UI. emit() applies the event to BridgeServer.localState and broadcasts `.event(...)` to the app's observer client. AppModel.applyTrackedEvent (AppModel.swift:1461) calls state.apply(event); SessionState.apply(.permissionRequested) sets session.phase = .waitingForApproval and stores session.permissionRequest (SessionState.swift:115-125). IslandSurface.notificationSurface(for:) maps it to a .sessionList notification card (IslandSurface.swift:23-34); OverlayUICoordinator.presentNotificationSurface opens the notch and plays the sound (OverlayUICoordinator.swift:321-332). The card shows title/summary/working dir and Allow/Deny buttons rendered from PermissionRequest.

5. User approves/denies in the notch. The card's buttons call AppModel.approvePermission(for:approved:) / approvePermission(for:action:) / approveFocusedPermission (AppModel.swift:1271-1394). These (a) optimistically mutate local state via state.resolvePermission(...), and (b) call send(.resolvePermission(sessionID, resolution: .allowOnce()/.deny(...)), ...) which dispatches the command over the app's LocalBridgeClient back to the SAME in-process BridgeServer. resolution is built by permissionResolution(for:) (AppModel.swift:1453): approved→.allowOnce(), denied→.deny(message:"Permission denied in Open Island.", interrupt:false).

6. Bridge resolves + replies down the held socket. BridgeServer.handle(.resolvePermission) (BridgeServer.swift:330) finds the matching pending entry. For Claude: resolvePendingClaudeInteraction (L2402) builds a `ClaudeHookDirective.permissionRequest(.allow(updatedInput, updatedPermissions))` or `.deny(message, interrupt)`, emits a follow-up activityUpdated/sessionCompleted event, and calls `send(.response(.claudeHookDirective(directive)), to: pendingInteraction.clientID)` — writing the directive line down the still-open hook socket (send → writeAll, BridgeServer.swift:2573-2584). Codex path: resolvePendingApproval → .codexHookDirective(.permissionRequest(.allow/.deny)) (L2377). OpenCode → .openCodeHookDirective; Cursor → .cursorHookDirective.

7. Hook unblocks + tells the agent. BridgeCommandClient.send returns that BridgeResponse to the CLI. The CLI runs ClaudeHookOutputEncoder.standardOutput(for: response) (ClaudeHooks.swift:636) which encodes the directive into the precise JSON Claude Code expects ({"continue":true,"suppressOutput":true,"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"/"deny", ...}}}) and writes it to FileHandle.standardOutput (OpenIslandHooksCLI.swift:84-86). The agent reads its hook's stdout and applies the permission decision. Cursor writes the directive JSON directly to stdout (OpenIslandHooksCLI.swift:98-103).

MECHANISM SUMMARY (quoting the answer to 'by what means'): approval is returned NOT by keystroke injection and NOT by a response file — it is delivered by holding the agent's hook subprocess blocked on its Unix-socket read, then writing a JSON 'directive' response back down that socket; the hook prints the directive to stdout, which the agent's hook protocol consumes as the allow/deny decision. KeystrokeInjector (Warp tab cycling) and TerminalTextSender (tmux/Ghostty reply text) are unrelated to approvals; TerminalTextSender only sends free-text replies to already-completed sessions.

WATCH PARALLEL PATH: In applyTrackedEvent, if watchRelay is non-nil (AppModel.swift:1501), the same event is forwarded to WatchNotificationRelay.notifyEvent, which for permissionRequested/questionAsked builds a WatchPermissionEvent/WatchQuestionEvent and pushes it as SSE to paired phones (WatchHTTPEndpoint.pushEvent). A phone POSTs /resolution {requestID, action} → WatchHTTPEndpoint.onResolution → relay maps requestID→sessionID → relay.onResolvePermission(sessionID, approved) → AppModel.approvePermission, re-joining the exact same step-5 path.

**Integration notes:** TO PORT THE PERMISSION ROUND-TRIP into another host, you must extract the whole socket-blocking loop, not just the UI:
1. A hook CLI binary the agent invokes (OpenIslandHooks) that reads stdin, sends over a Unix socket, and BLOCKS on read for the reply, then prints the agent-specific directive JSON to stdout. The agent-side contract is agent-specific: Claude Code's `PermissionRequest`/`PreToolUse` hook reads stdout JSON with `hookSpecificOutput.decision` (allow/deny, updatedInput, updatedPermissions) — see ClaudeHookOutputEncoder. Codex/Cursor/OpenCode each have their own directive shapes (CodexHookDirective, CursorHookDirective, OpenCodeHookDirective).
2. A bridge server (BridgeServer) that keeps the hook's file descriptor in a pendingApprovals-style map keyed by sessionID and only replies when the user decides. The blocking timeout must be long (24h) for interactive events — short timeouts would auto-fail-open and run/deny the tool without the user.
3. The command/response wire types (BridgeCommand.resolvePermission, BridgeResponse.*HookDirective) and BridgeCodec NDJSON framing.
4. The reducer (SessionState.apply + resolvePermission/answerQuestion) and the AgentEvent enum.
Hooks 'fail open' (CLAUDE.md): if the bridge/app is down, BridgeCommandClient.send throws, the CLI logs to stderr and returns without writing stdout, so the agent proceeds with its normal in-terminal prompt.

NOTIFICATIONS: Open Island does NOT use UNUserNotification/system banners. 'Notifications' = the in-notch overlay card (IslandSurface.notificationSurface) plus an NSSound .aiff (NotificationSoundService). To add a new actionable card type, add an AgentEvent case, handle it in SessionState.apply, map it in IslandSurface.notificationSurface, and (if it should reach the watch) add a WatchSSEEvent case + handling in WatchNotificationRelay.notifyEvent.

WATCH RELAY IS OPTIONAL: gated by AppModel.watchNotificationEnabled (UserDefaults key 'watch.notification.enabled', default false, AppModel.swift:433-444). startWatchRelay lazily constructs WatchNotificationRelay/WatchHTTPEndpoint only when toggled on; toggling off calls stopWatchRelay. Nothing in the local permission round-trip depends on it. To add a remote channel for a new event, extend WatchSSEEvent + notifyEvent.

BridgeServer.stopLocked() clears all pending maps and cancels client read sources, abruptly unblocking hooks with EOF (bytesRead==0 → returns nil → no directive → agent falls back to its own prompt).  • resolvePermission matching is keyed solely by sessionID, and only one pending interaction per session is stored per map (pendingClaudeInteractions[sessionID]). Parallel subagent permission prompts in the same session can overwrite each other; the WatchNotificationRelay code comments explicitly note this fan-out problem and that actionableStateResolved must clear ALL pending requests for a session (WatchNotificationRelay.swift:83-102).  • Optimistic UI: AppModel mutates local state.resolvePermission immediately, then sends the bridge command async (fire-and-forget Task in send(_:userMessage:)). If the bridge command fails to deliver, the notch shows resolved but the agent stays blocked until timeout.  • WatchHTTPEndpoint is an unencrypted plain-HTTP server on the LAN with only a 4-digit pairing code (10^4 space, 2-min expiry) issuing bearer tokens; SSE/resolution traffic (including command text/working dirs) is cleartext over TCP. Anyone on the network who guesses the code or sniffs a token can approve/deny agent permissions remotely. It is off by default, which mitigates this.  • Directive JSON shape is tightly coupled to each agent CLI's hook-output contract (Claude's hookSpecificOutput.decision, Codex/Cursor/OpenCode equivalents). An upstream CLI change to the hook output schema silently breaks approvals.  • KeystrokeInjector/TerminalTextSender depend on fragile AppleScript/AX paths (Warp menu item names, Ghostty terminal matching) and TCC grants that are invalidated by unsigned dev rebuilds (per CLAUDE.md) — but these are jump/reply features, not approval-critical.

### Open Island — "Jump Back to Terminal/IDE" + Process Discovery

**Summary:** Open Island maps a Claude Code (or Codex/Gemini/etc.) session to a concrete terminal window/tab/pane and brings it to the foreground using a per-terminal strategy: AppleScript for Terminal.app/iTerm2/Ghostty, native CLIs for WezTerm/Kaku/tmux/Zellij/VS Code/JetBrains, a Unix-socket JSON-RPC for cmux, a URL scheme for Codex.app, and a hybrid Accessibility-menu-click + read-only SQLite loop for Warp. Sessions are discovered by shelling out to `/bin/ps` and `/usr/sbin/lsof` (not sysctl/proc APIs), then walking the parent-PID chain to identify the host terminal. The only macOS permissions that matter are Automation/AppleEvents (osascript control of terminals + System Events) and Accessibility (Warp menu-item clicks); Full Disk Access is never needed because Warp's SQLite lives in the user's own Group Container.

**Data flow:** 1) DISCOVERY: ProcessMonitoringCoordinator.startMonitoringIfNeeded() (ProcessMonitoringCoordinator.swift:56) spins a Task that every 2s (:84) detaches to .utility and calls ActiveAgentProcessDiscovery.discover() (ActiveAgentProcessDiscovery.swift:58). discover() shells `/bin/ps -Ao pid=,ppid=,tty=,command=` (:194), classifies each process (isClaudeProcess :681, isCodexProcess :569, etc.), and for matches runs `/usr/sbin/lsof -a -p <pid> -Fn` (:481) to read cwd (fcwd->n line :484) and the open transcript .jsonl (which yields the session UUID via firstUUID :506). Host terminal app is inferred by walking parentPID up the process table and string-matching command paths (terminalApp :362 / recognizedTerminalApp :384). If no GUI terminal is found but a TTY exists, tmux CLI is queried (resolveTmuxInfo :768) to get session:window.pane + host terminal + socket.
2) ENRICHMENT: In the same detached block, TerminalJumpTargetResolver.resolveJumpTargets (TerminalJumpTargetResolver.swift:46) and TerminalSessionAttachmentProbe snapshots run AppleScript (`/usr/bin/osascript`) against Ghostty/Terminal/iTerm to enumerate live terminals/tabs/sessions, plus `wezterm cli list` / `tmux list-panes` for those families, matching by sessionID/tty/cwd/title and writing corrected JumpTarget fields back into SessionState.
3) WARP MAPPING (hook-time): When a Warp agent fires a hook, ClaudeHooks.defaultWarpPaneResolver (ClaudeHooks.swift:962) first does PID correlation — WarpProcessResolver.resolveCurrentPaneContext() (WarpProcessResolver.swift:39) walks getppid to the shell child of Warp's terminal-server, then WarpSQLiteReader.lookupPaneUUIDByShellPID enumerates terminal-server children via `pgrep -P` (WarpSQLiteReader.swift:363) and indexes Warp's tabs/pane_nodes/terminal_panes (:282) — and falls back to cwd lookup (lookupPaneUUID(forCwd:) :67). The resulting warpPaneUUID is stored on the JumpTarget.
4) JUMP: User clicks Jump -> AppModel.jumpToSession (AppModel.swift:1299) -> jump(to:) (:1308) detaches and calls TerminalJumpService().jump(to:) (TerminalJumpService.swift:245). jump() first handles tmux (switch-client/select-window/select-pane via tmux CLI :564, then the host terminal's own focus), then dispatches on resolvedBundleIdentifier (:330): iTerm/Ghostty/Terminal via AppleScript; Warp via jumpToWarpPane (:1108); WezTerm/Kaku via `wezterm cli activate-pane`; VS Code/JetBrains via their CLI; cmux via Unix socket; Codex.app via codex:// URL; Zellij via zellij CLI. All app activation ultimately uses `/usr/bin/open -b <bundleID>` (defaultOpenAction :1235).
5) WARP CYCLING: jumpToWarpPane (TerminalJumpService.swift:1108) opens Warp, polls NSWorkspace.frontmostApplication until Warp is frontmost (:1133), reads currentFocusedPaneUUID() from SQLite, and if it doesn't equal target.warpPaneUUID, repeatedly clicks the Tab>Switch to Next Tab menu item via Accessibility (DefaultKeystrokeInjector.sendCmdShiftRightBracket KeystrokeInjector.swift:59) — re-reading SQLite after each 0.1s settle until match or tabCount+2 attempts.
6) NOTIFICATION SUPPRESSION: ForegroundTerminalSessionProbe.matches (ForegroundTerminalSessionProbe.swift:71) compares NSWorkspace.frontmostApplication + the AppleScript-reported focused surface to skip jump notifications when already focused.

**Integration notes:** TO PORT "jump back to terminal/IDE" into another host you need three layers:

A) The dispatcher + per-terminal strategies — TerminalJumpService.swift is almost self-contained. It depends only on: a JumpTarget model (OpenIslandCore/AgentSession.swift:138 — fields terminalApp, workspaceName, paneTitle, workingDirectory?, terminalSessionID?, terminalTTY?, tmuxTarget?, tmuxSocketPath?, warpPaneUUID?, codexThreadID?), the KeystrokeInjector protocol (KeystrokeInjector.swift), and the WarpSQLiteReader (WarpSQLiteReader.swift). To port: copy TerminalJumpService.swift + KeystrokeInjector.swift + WarpSQLiteReader.swift + WarpProcessResolver.swift + the JumpTarget struct.

B) Producing a correct JumpTarget — you must supply terminalApp + at least one precise locator. The locators come from (i) agent hooks at session start (ClaudeHooks.withRuntimeContext :947 infers terminalApp from env, sets warpPaneUUID, CMUX_SURFACE_ID, Zellij pane), and (ii) live enrichment via ActiveAgentProcessDiscovery + TerminalJumpTargetResolver + TerminalSessionAttachmentProbe. If you only need jump (not Open Island's full attachment state machine), you can port just ActiveAgentProcessDiscovery (ps+lsof+parent-walk) to get cwd/tty/terminalApp, and TerminalJumpTargetResolver to upgrade tty->sessionID via AppleScript.

### Multi-Agent & Multi-Terminal Abstraction (Open Island / OpenIslandCore)

**Summary:** Open Island supports 10 agents but there is NO generic agent/terminal protocol — agents are enumerated by a single `AgentTool` enum and dispatched through hardcoded per-agent switch arms in the bridge, hook CLI, transport, and installation coordinator. The "abstraction" is really (a) a shared, agent-agnostic session model + reducer + event enum + Unix-socket bridge, plus (b) one concrete code path per *hook protocol family*. There are only 5 distinct hook protocols (Claude, Codex, OpenCode, Cursor, Gemini); the 5 "Claude Code forks" (Qoder/Qwen/Factory/CodeBuddy/Kimi) carry almost no dedicated code — they reuse Claude's payload/installer parameterized by directory + source string. Codex additionally has a second, entirely different integration (CodexAppServer, a JSON-RPC app-server client) layered on top of its hooks. A Claude-only subset is very cleanly extractable.

**Data flow:** Runtime (per CLAUDE.md and verified in code): `agent hook fires → OpenIslandHooks CLI (stdin JSON + --source)` → OpenIslandHooksCLI.swift:32 main() decodes by source: isClaudeFormat sources (claude/qoder/qwen/factory/droid/codebuddy/kimi) decode ClaudeHookPayload and send `.processClaudeHook` over the Unix socket; codex/cursor/gemini have their own decode+command. → BridgeCommandClient/BridgeTransport encodes NDJSON → BridgeServer.handle(_:from:) (BridgeServer.swift:302) switches on the command to one of 5 hardcoded handlers (handleClaudeHook:614 etc.). The handler builds AgentEvent values and calls emit() (2564), which applies to localState and broadcasts `.event` envelopes over the socket. → AppModel's LocalBridgeClient (AppModel.swift:516) consumes the AsyncThrowingStream<AgentEvent> and calls state.apply(event) (AppModel.swift:1486) — SessionState.apply (SessionState.swift:56) is the single reducer for every agent. UI observes SessionState. For interactive permission/question, the app sends `.resolvePermission`/`.answerQuestion` back through the bridge; BridgeServer matches the per-agent pending-interaction map (pendingClaudeInteractions etc., BridgeServer.swift:69-72) and writes the agent-specific directive back to the still-blocked hook process (hooks fail open if the app is down). On launch (SessionDiscoveryCoordinator.loadStartupDiscoveryPayload:83): each agent's registry is loaded + its transcripts/rollouts discovered in parallel, then reconciled with live processes (ProcessMonitoringCoordinator) before the live bridge takes over. **Dependencies:** Swift 6.2 / swift-tools-version 6.2; macOS 14+ (platforms .macOS(.v14)); Foundation, Dispatch, Darwin (raw AF_UNIX sockets in BridgeServer/LocalBridgeClient — no networking framework); OpenIslandCore is a pure library with ZERO external SPM deps (Package.swift target has no dependencies array); OpenIslandApp deps (NOT needed for core extraction): swift-markdown-ui (MarkdownUI) for rendering, Sparkle for auto-update; SwiftUI + AppKit (app shell only; AppModel uses @Observable); Unix domain socket at BridgeSocketLocation.defaultURL (+ legacy /tmp path) — local-first, no server, no network host; Reads agent config dirs on disk: ~/.claude/settings.json (+ ~/.qoder, ~/.qwen, ~/.factory, ~/.codebuddy for forks, ~/.kimi/config.toml for Kimi, ~/.codex for Codex) and JSONL transcripts under the Claude projects dir; Optional: Accessibility/Automation TCC entitlements for precision terminal jump (TerminalJumpService, KeystrokeInjector) — independent of the agent abstraction

**Risks:** • No agent/terminal PROTOCOL exists — `AgentTool` (AgentSession.swift:3) and `AgentIdentifier` (AgentHookIntent.swift:19) are two separate hardcoded enums that can drift; adding/removing an agent means editing ~6 hardcoded switch sites (BridgeTransport, BridgeServer, HooksCLI, HookInstallationCoordinator, SessionDiscoveryCoordinator, AppModel). Extraction must prune ALL of them or it won't compile.  • Two overlapping agent enums (AgentTool has geminiCLI/kimiCLI/qwenCode; AgentIdentifier has gemini/kimi/qwenCode + claudeUsageBridge). Raw values are stable UserDefaults keys — do not rename when trimming.  • AgentEvent leaks 4 per-agent metadata cases (claude/gemini/openCode/cursor) into the otherwise-shared event enum — Codex/forks ride the generic sessionMetadataUpdated, so the metadata model is inconsistent across agents.  • Claude forks (Qoder/Qwen/Factory/CodeBuddy) and Kimi share Claude's RUNTIME payload (ClaudeHookPayload) — Kimi only differs in install format (TOML). So 'Claude-only' code is nearly identical to '6-agent' code at runtime; the savings of dropping forks are almost entirely in install/UI config, not the hot path.  • Codex is the outlier: it has BOTH a hook path (CodexHooks) AND a JSON-RPC app-server path (CodexAppServer/CodexSessionTracking, ~3000 lines). If a future port wants Codex it is far more work than any other agent. Irrelevant for Claude-only but a major asymmetry in the 'modularity' story.  • BridgeServer.swift is a 2681-line monolith mixing shared socket transport with all 5 per-agent handlers — extracting Claude requires surgical deletion inside one large file, not just dropping files.  • HookInstallationCoordinator and SessionDiscoveryCoordinator (App layer) duplicate per-agent restore/persist logic inline; trimming risks leaving dangling status accessors / @Observable properties referenced by UI.
